Date Wed 03 September 2014 By laoyi Category Python. Tags Python web tool

问题描述

博主购买了某shadowsocks服务， 众所周知shadowsocks的配置文件外貌如下：

{
  server:server_name,
  server_port:1234,
  local_port:1234,
  password:server_password,
  method:encypt_method,
  timeout:600
}

博主的服务器列表在某网页上， 因此每次需要配置都要到网页上找。

而比较麻烦的是， 随着gfw的不断升级和修改配置， 服务器的可用性也一直变化， 大部分服务器在一段时间后都会变得不可用， 而且由于端口和密码的改变， 也经常需要修改本地的ss配置文件以保持成功连接。

因此原有情况博主需要经常登录网站获得服务器列表， 手动ping每个ip查找可用的服务器， 再把服务器连接信息写到本地的config.json文件中， 最后重新启动shadowsocks。

相当痛苦。

需求描述

一个本地的自动化工具， 自动抓取最新的服务器列表， 自动分析提取出服务器配置信息， 自动测试与服务器的连通性和延迟时间， 最后自动生成最优连接服务器的配置文件输出到config.json。

这样博主只需要拿生成的config.json给shadowsocks使用即可。

另外， 由于多个服务器的连接测试都需要消耗一定的时间， 因此最好能够并发检测， 最后统计并处理所有结果。

Date Wed 16 July 2014 By laoyi Category web. Tags web tool

最近找到了许多好用的Web工具，在此记录一下使用心得。

爬墙类

几年前， 詹神曾经在班级邮件里说过， “作为程序员，翻墙应该是基本技能”。 这句话我一直印象很深。

当时的自己年少无知， 对爬墙技术的了解也不深， 但随着对Google依赖的不断深化， 搜文档、找工具、查资料， 加上许多信息工具的应用， 这才越来越体会到这句话的深刻含义。

爬墙的一个原则是不要在一棵树上吊死， 这句是在某知名博主的博客上看到的， 永远要留有备用的手段。

goagent

这货连接的是Google的服务器， 凭借限定流量免费的Google App Engine服务， 热心的技术人员开发出了一套转发代理的工具， 是最近一段时间比较稳定的爬墙方式。

自己搭建服务器的话需要注册一个GAE的账户， 建立一个新的App， 然后把服务器端给传上去。

本地的客户端通过指定AppID连接服务器端， 把本地的网络请求转发给服务器， 由服务器代理处理页面并返回结果。

安全问题

月光博客之前曾经谈过这货的安全性问题， 由于CA证书是伪造的， 而且默认的CA证书是公开的， 因此非常容易被中间人攻击。

解决的方案是把目录下的默认CA.crt删掉， 让GoAgent启动时自行生成新的唯一证书， 相当于私有的证书。 然后再把这个新的CA导入到系统的证书信任里。

此外， 最好开启服务器端的验证， 在配置文件里把validate开启。

但不管怎样优化也不能让这货有比较好的安全性， GAE在转发用户请求的时候会把AppID也写入到User Agent里面去， 这非常容易暴露用户的身份。

玩转莱特币：使用CPUMiner和proxy挖矿的办法

Date Mon 16 June 2014 By laoyi Category web. Tags litecoin

挖矿分为两种， 简单暴力的直接挖矿， 以及加入矿池进行合体挖矿。

由于现在的全网算力已经超乎想象(博主玩的时候大概为300GH/s)， 没有专用矿机只能选择矿池挖矿。

由于博主以娱乐为主， 没有钱和精力购买高速专用矿机， 甚至没有好的显卡拿来用GPU挖矿， 因此使用的是简单的CPU挖矿方式(10KH/s......)。

如果正儿八经的想赚点钱， 推荐至少要弄个好的显卡用GPU挖矿。

在网上有一些算力的统计表， AMD系列的显卡跑GPU挖矿一般可以达到100KH/s以上。

矿池的选择

由于博主用的是大陆的局域网， 选择矿池显得比较困难。

首先需要排除掉那些没什么人的矿池， 听说没什么人的矿池由于全池算力都很差， 因此找到新块的几率非常低， 进而导致每个矿工分到的收益也很小。

此外， 由于网络的问题， 需要选择能够稳定连接的矿池， 否则经常断线就只能白白消耗电能。

现在的矿池基本上都支持stratum协议， 这种协议本质上是建立一个代理服务器中转矿机和目标服务器的连接， 由于有了这种类似于buffer的协议层， 相比最初的getwork方式直连， 不仅网络的带宽占用变小， 连接的稳定性等各方面也有较好的提升。

许多矿池甚至强制性使用stratum协议， 例如coinotron。

矿池的选择可以参考莱特币的info页面：

https://litecoin.info/Mining_pool_comparison

玩转莱特币：概念

Date Sun 08 June 2014 By laoyi Category web. Tags litecoin

关于p2p货币

其实这个概念已经出现好几年了， 作为一种新类型的虚拟货币， 它们的共同点是去中心化。

发行

传统的货币必须要有一个中央发行机构， 负责印发钞票，对该货币的市场拥有控制的能力。 据说这种形式的货币普遍存在的不足是容易受到各种势力的管制。

比特币及各种类型的p2p货币(莱特币、狗币...)则不是如此， 他们没有中央控制机构， 而是基于特定的困难算法， 通过一定的速率产生。 没有任何机构能够控制新币的产生， 这样就有效防止了受到某些势力管制的危险。

交易记录

而对于交易记录， 传统的货币是通过某些中央机构进行存储， 比如银行的数据库， 仍然是中心化的特点。 这种特点导致修改交易记录成为可能， 例如直接篡改数据库内容。

与此相反， p2p货币的交易记录存储在全网的所有运行的机器中， 相当于分布式的数据库。 这样， 如果某人想要篡改某一次的交易记录， 则需要入侵网络中所有的计算机， 基本上不可能。

交易确认

使用一种叫做块链的东西用于储存交易记录， 每份交易储存在区块中， 区块逐个相连形成块链。

通过困难的计算产生区块， 当新的交易产生时， 数据写入区块中， 区块插入块链。 这样就完成了交易的记录。

此外， 由于没有机制使区块从块链中移除， 因此这种类型的交易是不可逆的。

实习求职：一些面试经验的总结(简历篇)

Date Sun 11 May 2014 By laoyi Category Life. Tags audition recruitment

转眼间已经五月中旬， 最近的一个多月先后面试了几家公司， 顺利拿到了百田和腾讯的实习offer。

由于比较忙， 而且腾讯的面试略微零碎， 因此没有总结出完整的记录。

不仅在自己的经历中收获了很多东西， 在和朋友们的交流中也听说了很多东西， 因此写一篇通用的总结经验， 留给以后温故知新。

关于简历

简历是一个大问题， 许多人因为简历没有写好导致直接在筛选的时候就被干掉， 连笔试机会都没有。

眼下的简历一般分为两种， 自己制作后通过各种方式提交的自制简历， 电子版也好印刷版也好， 普遍存在于各种行业中的中小型公司。

另外一种是公司提供的表格进行填写， 一般是直接给个网址， 通过表单录入， 例如腾讯就是。 这种方式需要一定的技术支持， 所以在小型创业型公司比较少见。

以下先以自制型简历记录一般做法。

自制型简历特点是自由度非常大， 想怎么搞都行， 想要记录的内容任由制作者自行决定。

因此首先需要做的是保证简历的清爽和清晰。

不使用边框是比较好的， 因为边框的加入会导致整体效果比较死板， 每一个项目之间都加入边框整个简历变得密密麻麻， 阅读者也难以聚焦; 而如果是只在外层包一个大边框更显得画蛇添足。

整体排版以简单的纵向块分隔为主， 若干行组成一个块， 每一块中记录某个方面的信息。

每个块给个标题表明内容， 例如：个人信息、教育背景、获奖经历等。

个人信息

最佳实践是在简历的最顶端给个大大的姓名， 接下来的第一个块留给个人信息和联系方式， 这样的好处是非常直观， 便于hr确定身份联系发offer。 (^_~)

Android基础学习（一）：基本概念与理念

Date Thu 01 May 2014 By laoyi Category Mobile. Tags Android

眼下正是移动客户端应用泛滥的时代， Android与iOS无疑是当前移动客户端最为热门的两大系统。

于是博主也跟随潮流， 学起了Android。

本博文不涉及具体代码， 只简单记录整理基本概念， 权当总结。

Android应用的基本理念

Android应用与传统应用程序一样基于操作系统提供的抽象硬件环境， 而且有更多的运行限制。

由于Android系统基于Linux内核， 存在着众多的保护机制， 每个Android应用会通过各自独立的用户启动和运行。 并且每个应用在运行时处在Android系统内建的沙盒中， 与外界多了一层隔离。

在资源管理方面， Android应用必须声明对特定资源的访问请求， 才可执行对应的操作。 例如外部存储器的读写、 照相机的调用、 网络的调用等。 这些权限声明在用户安装的过程中进行确认， 仅当用户授权才可以安装应用。

虽然这点在实际操作过程中有很大变数，哈哈～

与桌面程序较为不同的是， Android应用更多依赖于用户的交互，

在Android应用中事件是一个非常重要的概念， 用户的任何操作都会触发一个事件， 例如手指的滑动，或者是按钮的点击。

当用户的事件触发时， Android系统通知应用程序事件的发生， 交由应用进行处理。

在Android应用的内部， 通过编写对应的回调函数来完成对事件的处理, 例如onClick()函数， 当用户点击按钮时该回调函数就会被调用。 这也就构成了最简单的Android应用形式。
