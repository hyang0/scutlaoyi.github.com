Git 与 GitHub 不完全教程（二）

Date Sat 07 September 2013 By laoyi Category git. Tags git github

分支

分支(branch)是Git的一个强大功能，它可以在不改变现有仓库、不新增仓库的情况下，通过新建一个分支以建立当前文档一模一样的支线版本，在支线版本的修改不影响主线文档的内容。

分支的一个重要的功能是能够在同一个时间点上保存多份不同版本的代码，并且均为随时可用的状态，根据官方的一个示例，你可以保持一份稳定的发布版，在发布版内的代码都是可运行的，之后，你可以再拥有一个分支专门用于开发，在这上面的代码用于任意修改和处理，不对外，还能再有一个分支用于测试开发新的功能。这三个分支都可以随时切换随时合并，应变自如。

当支线任务成功完成时，可通过命令简单地将支线版本合并到主线上，若由于手潮不幸把事情搞砸了，可直接删除支线，回到主线上，则原有的文档内容不受任何影响。这种功能特别适合在当前状态下需要测试开发一些新的功能，但又想要与主线的开发有所并行的情况。

默认情况下，git创建完仓库后会生成一个主分支master，所有的操作均在主分支上进行，这就相当于文档的主线版本。而我们可以按自己的需要增删改查新的分支。

分支的创建和切换

显示当前仓库的分支状态:

git branch

Git会列表显示当前的所有分支，并在当前所在分支上作相应标记。 新建分支（假设需要新建的分支名字为:newBranch）:

git branch newBranch

此时再次检查branch状态就会发现新的分支已经出现了。 新的分支和现有的分支内容完全相同，相当于当前分支的一个副本。 跳转到某个分支:

git checkout newBranch

即可。

分支的删除

git branch -d XXX

完成了分支的转换后，分支内的操作与前文的基本操作一致。

分支的合并

1.最简单的合并

新建的分支在原来分支的基础上出现了新的变更，而原来的分支没有变化。此时的合并操作只需要直接把新分支的内容合并到原有分支即可。

假设有两个branch: master && newChange, newChange从master而来，newChange发生了一次新的提交，而master没有变化。此时需要将newChange合并到master上，需要做的是切换到master分支，然后执行如下指令:

git merge newChange
2.比较复杂的合并

新建的分支和原来的分支均产生了更改，然而更改的内容没有冲突，即修改了原有内容的不同部分或者根本没有修改到同一个文件，这种情况下Git会自动进行处理，将新的修改自动合并到分支中来。使用的命令与前面的命令一样。

所产生的变化是，这种情况下合并操作类似commit，我在测试的时候Git提示需要输入合并的说明信息。合并后调用git log发现多了一次merge记录，与commit操作一样存在记录。

3.最后一种最复杂的情况

不仅两个分支产生了更改，还更改了相同的部分，这就产生了合并的冲突，Git的自动合并失败，此时需要做的是自己处理冲突并再次提交。

假设当前有分支branchOne && branchTwo，均从master分支而来，master分支原有一个readme文档。branchOne && branchTwo各自在文档的最后添加了一行新的不同的内容，并各自提交。此时，在branchTwo分之下调用 “git merge branchOne”，Git会提示冲突出现:

Auto-merging readme.txt
CONFLICT (content): Merge conflict in readme.txt
Automatic merge failed; fix conflicts and then commit the result.

在这种情况下，打开提示冲突的文档，你会发现所有冲突的地方Git均做了标记，HEAD后的内容为当前分支的版本，”======”分割后下方的为合并方的版本，需要做的是删除所有标记，保留自己想要的版本。在处理完所有冲突后，再次add 和commit即可。

远端仓库（remote）

远端仓库，就是架设在其他机器上的Git仓库，通过互联网连接后可通过Git获取和修改远端仓库的数据，达到协作的目的。比较有名的免费公有仓库是GitHub： https://github.com

在此以GitHub为例记录Git中关于远端仓库的若干使用步骤。

假设需要使用的远端仓库为我的操作系统大作业仓库： https://github.com/scutLaoYi/operating-system-solution.git

从远端仓库直接将仓库复制到本地

执行如下命令:

git clone https://github.com/scutLaoYi/operating-system-solution.git

成功完成后在当前目录下就会出现一个文件夹里面存放了该仓库。

获取当前的远端仓库列表

进入仓库目录，使用命令:

git remote

如果使用的是clone命令复制仓库，则存在一个origin的remote。否则remote得到空的仓库列表。

将远端仓库添加到remote列表中

git remote add newRemote https://github.com/scutLaoYi/operating-system-solution.git

此时将该仓库地址添加到remote列表中，名字为newRemote.

使用pull命令从远端仓库拉取新版本

git pull newRemote master:another

此句将远端仓库newRemote中的master分支拉取到本地的another分支中。这里要注意分支的顺序，前面的是远端仓库的分支名，冒号后的为本地分支的名字。

推送本地仓库到远端

git push newRemote master:remoteBranch

向newRemote远端仓库的remoteBranch分支推送本地仓库的master分支。

需要用户名和密码完成认证。自行测试的时候貌似还认证了邮箱。

杂七杂八的破玩意儿

最后聊一下Git工具的proxy连接，由于千奇百怪的网络问题，很容易导致Git与GitHub服务器的连接失败，这种情况下如果手头有一些生龙活虎的proxy工具，可以通过其跳转使Git连接到GitHub服务器，完成工作。

本人使用的是简单的http连接方式，Git配置http的proxy十分方便，如下命令即可:

git config –global http.proxy XXX.XXX.XXX.XXX:XXXX

XXX里填的是proxy的地址和端口号 某些工具需要ssl证书，简单地可以直接关闭Git的证书验证:

git config --global http.sslVerify false

或者通过以下命令指定证书的路径（此方法未测试）:

git config --global http.sslCAinfo XXX/XXX/XXX

至于proxy的相关内容不在本帖讨论范围。
